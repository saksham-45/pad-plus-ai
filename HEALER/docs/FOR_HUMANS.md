# HEALER — модуль самодиагностики и самовосстановления

## Где и как применять HEALER?

HEALER подходит не для всего подряд, а для проектов с определёнными признаками. Вот сценарии, где он реально полезен.

---

### 1. Пет-проекты и стартапы (одна кодовая база)

**Пример:** вы пишете Telegram-бота, веб-сервер на FastAPI или скрипт для парсинга. Проект один, всё в одной папке.

**Как применить:**
1. Скопировать папку `aethon/` (X-RAY ядро) в свой проект
2. Вставить 2 строки в код: `start_trace("request")` в начале обработки, `trace.end()` в конце
3. Запустить `python -m healer.diagnostics.runner --path ./data/trace_store`

**Что HEALER найдёт:** незакрытые файлы, медленные импорты, мёртвый код, ошибки без обработки.

**Почему подходит:** zero dependency — не нужно устанавливать ничего, кроме Python. Просто скопировал и пошло.

---

### 2. Продакшн-системы (наблюдение в реальном времени)

**Пример:** ваш сервер обрабатывает тысячи запросов в день. Нужно знать, что происходит внутри, и автоматически реагировать на проблемы.

**Как применить:**
1. Встроить X-RAY во входные точки вашего приложения:
   - При HTTPS-запросе — создать trace
   - При вызове внешнего API — создать span
   - При работе с базой данных — создать span
2. Включить `--watch` режим HEALER:
   ```bash
   python -m healer.diagnostics.runner --watch --interval 60 --fail-on error
   ```
3. Или запустить HTTP API:
   ```bash
   python -m healer.api --port 8090
   # Теперь можно дёргать /api/v1/run из cron или CI
   ```

**Что HEALER найдёт:** аномалии задержек (один запрос работает дольше остальных), утечки (spans, которые не завершаются), нарушения причинности (вызовы в нелогичном порядке).

**Почему подходит:** X-RAY ядро не требует async, не блокирует основной поток, пишет на диск асинхронно. Отказоустойчивость — при падении HEALER приложение продолжает работать.

---

### 3. CI/CD пайплайн (автоматическая проверка перед деплоем)

**Пример:** перед выкаткой новой версии в продакшн нужно проверить, не появились ли регрессии.

**Как применить:**
```bash
# В CI скрипте (GitHub Actions, GitLab CI, Jenkins):
python -m healer.diagnostics.runner --quiet --fail-on error --output healer-report.json

# Если exit code = 1 — пайплайн останавливается
```

**Что HEALER сделает:** прогонит 7 детекторов, найдёт проблемы, вернёт JSON-отчёт. Если есть ошибки уровня `error` или `critical` — пайплайн упадёт.

**Почему подходит:** не требует установки зависимостей в CI-окружении. `--quiet` даёт чистый JSON, который можно распарсить любым скриптом.

---

### 4. Self-healing: когда HEALER чинит сам себя или соседний проект

**Пример:** вы запустили HEALER на проекте `my-app`, он нашёл и починил 3 проблемы. После этого HEALER может продиагностировать сам себя (`healer-viewer`) и починить свой же код.

**Как применить:**
1. Запустить HEALER на целевом проекте:
   ```bash
   python -m healer.diagnostics.runner --path ../my-app/data/trace_store
   ```
2. Запустить healer-viewer (веб-дашборд), который тоже инструментирован X-RAY:
   ```bash
   cd ../healer-viewer
   start.bat
   ```
3. Viewer может запустить диагностику на себе — HEALER найдёт проблемы в viewer.py и применит патчи

**Что HEALER найдёт:** проблемы как в чужом, так и в своём коде. Например, незавершённый span в `viewer.py` — HEALER обернёт его в `try/finally`.

**Почему подходит:** TraceStoreRegistry изолирует трейсы разных проектов. HEALER не путает данные `my-app` и `healer-viewer`.

---

### 5. Монолит с несколькими модулями

**Пример:** у вас большой проект с модулями `auth`, `payment`, `notifications`. Каждый модуль пишет свои трейсы.

**Как применить:**
```bash
# Проверить каждый модуль отдельно
python -m healer.diagnostics.runner --path ./data/auth/trace_store --output auth-report.html
python -m healer.diagnostics.runner --path ./data/payment/trace_store --output payment-report.html
python -m healer.diagnostics.runner --path ./data/notifications/trace_store --output notifications-report.html
```

**Что HEALER найдёт:** для каждого модуля — свои проблемы. В модуле `payment` может быть утечка ресурсов, в `auth` — медленный импорт, в `notifications` — мёртвый код.

**Почему подходит:** store изолирован по папке, можно параллельно запускать диагностику на разных модулях.

---

### Когда HEALER НЕ нужен

- **Если у проекта нет X-RAY** — HEALER ничего не проанализирует, нужны трейсы на диске
- **Если проект на JS/TS без Python** — HEALER умеет патчить JS, но для диагностики нужны трейсы, которые пишет X-RAY (только Python)
- **Если проект одноразовый** — скрипт для разового импорта данных, который запустили и забыли. HEALER нужен там, где код живёт долго
- **Если проект встроенный (embedded)** — HEALER требует Python 3.12+ и файловую систему для хранения трейсов

---

### Таблица: проект → что HEALER даёт

| Тип проекта | Что HEALER найдёт | Режим | Формат отчёта |
|-------------|-------------------|-------|--------------|
| Веб-сервер (FastAPI/Flask) | Зависшие запросы, утечки, аномалии | `--watch` | HTML в дашборд |
| Telegram-бот | Незакрытые соединения, медленные импорты | `monitor` | Таблица в консоль |
| CI/CD пайплайн | Регрессии, мёртвый код, ошибки без fallback | `--quiet` | JSON для парсинга |
| Self-healing viewer | Проблемы в собственном коде | `auto` | Лайв-лента в браузере |
| Монолит (N модулей) | Модуль-specific проблемы | Раздельные запуски | N HTML-файлов |
| Исследовательский проект | Аномалии, нарушения причинности | `suggest` | HTML + рекомендации |

HEALER — это программа, которая **сама следит за собой и чинит себя**. Представьте механика, который сидит внутри вашего проекта и смотрит, всё ли в порядке, а если что-то сломалось — сразу чинит.

## Что он умеет делать прямо сейчас?

### 1. Наблюдать (X-RAY ядро)

Система записывает каждое действие в программе: кто что вызвал, сколько времени работало, не упало ли с ошибкой. Это как **чёрный ящик в самолёте** — полная запись всего, что происходит.

### 2. Находить проблемы (7 детекторов)

HEALER автоматически проверяет ваш код и находит типовые ошибки:

- **Медленные импорты** — если программа загружается слишком долго
- **Ошибки без страховки** — если код упадёт, но никто это не обработает
- **Мёртвый код** — функции, которые написаны, но никогда не вызываются (пыль)
- **Утечки ресурсов** — если программа открыла файл, но забыла его закрыть
- **Нарушения причинности** — странный порядок выполнения, который не имеет смысла
- **Аномалии задержек** — если какой-то участок кода работает подозрительно долго
- **Незавершённые операции** — span'ы (отрезки работы), которые начались, но не закончились

### 3. Исправлять (AST-патчер)

Если HEALER нашёл проблему, он может **сам сгенерировать исправление** и применить его. Никакого ИИ, только точные преобразования кода:

- Перенести медленный импорт внутрь функции
- Обернуть код в `try/finally`, чтобы ресурсы точно закрылись
- Добавить таймаут к сетевым вызовам
- Удалить мёртвый код
- Переписать `open()` на безопасный `with`-блок

### 4. Проверять и откатывать

После исправления HEALER **запускает тесты**. Если тесты упали — он **откатывает изменения** и восстанавливает оригинальный код (у него есть бэкап).

### 5. Учиться на ошибках (Meta-обучение)

Система запоминает, какие исправления сработали, а какие нет. Со временем она **выбирает только то, что реально помогает**, и перестаёт предлагать бесполезные фиксы.

## Инструменты для работы

- **CLI** — командная строка: запустил, получил отчёт
- **Веб-дашборд (Viewer)** — отдельное приложение с интерфейсом для просмотра трейсов и диагностики
- **HTTP API** — можно дёргать программно (есть версионирование `/api/v1/`)

---

## Три режима работы

| Режим | Что делает |
|-------|-----------|
| **monitor** | Только смотрит. Найдёт проблемы и покажет отчёт, но ничего не трогает |
| **suggest** | Найдёт проблемы и покажет, как их можно исправить (но сам не применяет) |
| **auto** | Найдёт → починит → проверит тестами. Полная автоматика |

---

## Технические характеристики (сухие цифры)

- **121 тест** — все проходят
- **0 внешних зависимостей** — работает на чистом Python без установки пакетов
- **7 слоёв архитектуры** — каждый можно менять независимо
- **3 формата отчёта** — таблица в консоль, JSON для машин, HTML для браузера
- **Версионирование API** — `/api/v1/` не сломается при обновлении

## Как попробовать?

### Шаг 1. Откройте терминал

Запустите **командную строку** (cmd), **PowerShell** или **Git Bash**. Перейдите в папку HEALER:

```bash
cd HEALER
```

Все команды ниже выполняются из этой папки.

---

### Шаг 2. Демо-запуск

```bash
python main.py
```

**Кто запускает:** любой, кто хочет увидеть HEALER в действии — разработчик, тестировщик, тимлид.

**Зачем:** быстрая демонстрация. Скрипт генерирует учебные трейсы (как будто программа что-то делала), затем запускает диагностику и показывает отчёт. Это **единственная команда, где данные создаются автоматически** — для всех остальных нужны реальные данные.

**Что вы увидите:**
- Сначала зелёные сообщения — X-RAY записал несколько учебных трейсов
- Потом лайв-ленту детекторов — они проверяют эти трейсы один за другим
- В конце — сводную таблицу: сколько проблем найдено, какие severity, какие детекторы сработали

**Когда использовать:** когда хотите просто посмотреть, как выглядит работа HEALER. Не требует никакой подготовки.

---

### Шаг 3. Проверить свой проект

```bash
python -m healer.diagnostics.runner
```

**Кто запускает:** разработчик, который уже подключил X-RAY к своему проекту и хочет узнать, есть ли в нём проблемы.

**Зачем:** HEALER читает трейсы из папки `data/trace_store/` (рядом с HEALER) и прогоняет через 7 детекторов. Если у вашего проекта своя папка с трейсами — укажите её:

```bash
python -m healer.diagnostics.runner --path /путь/к/трейсам
```

**Что происходит под капотом:**
1. Загружаются трейсы с диска в изолированное хранилище (не трогает данные других проектов)
2. По очереди запускаются 7 детекторов — вы видите лайв-ленту:
   ```
     ▶ SpanAnalyzer...
       ⚡ [warning] Span server.request не завершён
     ✅ SpanAnalyzer — 12 reports — 2ms
     ▶ SlowImportDetector...
     ✅ SlowImportDetector — 0 reports — 0ms
   ```
3. В конце — сводная таблица по детекторам и severity

**Форматы вывода:**
| Флаг | Результат |
|------|-----------|
| (без флагов) | Красивая таблица в консоль с иконками |
| `--format json` | Сырой JSON — для парсинга другими программами |
| `--format html` | Самодостаточная HTML-страница — можно открыть в браузере |
| `--quiet` | То же что `--format json` |
| `--output report.html` | Сохранить отчёт в файл |

**Примеры для реальных сценариев:**
```bash
# Проверить конкретный проект
python -m healer.diagnostics.runner --path ../my-project/data/trace_store

# Сохранить HTML-отчёт и открыть в браузере
python -m healer.diagnostics.runner --format html --output report.html
start report.html   # Windows
open report.html    # macOS

# Для CI/CD: тихий JSON, ошибка если есть critical
python -m healer.diagnostics.runner --quiet --fail-on critical --output ci-report.json

# Непрерывный мониторинг (проверка каждые 30 секунд)
python -m healer.diagnostics.runner --watch --interval 30
```

**Когда использовать:** когда X-RAY уже подключён к проекту и пишет трейсы на диск. HEALER читает их и выдаёт диагностику.

---

### Шаг 4. Smoke test — убедиться, что всё работает

```bash
python scripts/smoke_test.py
```

**Кто запускает:** разработчик после изменений в HEALER, или любой, кто сомневается — не сломалось ли что-то.

**Зачем:** smoke test (дымовой тест) — это быстрая проверка **всех слоёв системы**. Он не заменяет 121 юнит-тест, но за 2 секунды показывает, что:

1. **X-RAY Kernel** — трейсы и спаны создаются, причинность проверяется
2. **Diagnostics** — `run_diagnostics()` возвращает отчёты, `DiagnosticReport` корректно создаётся
3. **Patch Engine** — PythonPatcher видит 5 паттернов, PatchResult умеет считать diff
4. **Verification** — результаты верификации создаются
5. **Meta-Learning** — MetaLearner записывает и загружает историю
6. **Orchestrator** — оркестратор создаётся, режимы переключаются

**Что вы увидите:**
```
=== HEALER Smoke Test ===

[1] X-RAY Kernel
  [PASS] Trace/Span creation
  [PASS] Causal validator

[2] Diagnostics
  [PASS] run_diagnostics() — 0 reports
  [PASS] DiagnosticReport creation

... (все 6 слоёв) ...

========================================
  SMOKE TEST: ALL PASSED
========================================
```

**Когда использовать:**
- **После каждого изменения** в HEALER — запустить и убедиться, что базовая логика не сломана
- **После `git pull`** — проверить, что обновлённая версия работает
- **Перед важной диагностикой** — убедиться, что система в порядке
- **Если что-то пошло не так** — smoke test либо укажет на сломанный слой, либо подтвердит, что проблема не в HEALER
