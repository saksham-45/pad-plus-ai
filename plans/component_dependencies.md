# 🔗 Зависимости компонентов NeuroMind AI

## 📋 Обзор

Этот документ описывает зависимости между компонентами системы NeuroMind AI. Понимание зависимостей критично для правильной архитектуры, тестирования и разработки системы.

## 🏗️ Архитектурные зависимости

### 1. Ядро системы (Core)

```
ANTI_DIRECTIVE (анти-директива)
├── Проверяет целостность системы
├── Является философским ядром
└── Не зависит от других компонентов
```

```
Pipeline Executor
├── Зависит от: ANTI_DIRECTIVE
├── Используется: Все обработчики запросов
├── Зависит от: Intent Router, Safety Layer, Truth Loop
└── Управляет: Порядком обработки запросов
```

```
Intent Router
├── Зависит от: Core
├── Используется: Pipeline Executor
├── Зависит от: Memory (для контекста)
└── Возвращает: Стратегию обработки
```

```
Safety Layer
├── Зависит от: Core
├── Используется: Pipeline Executor
├── Проверяет: Все входящие и исходящие запросы
└── Возвращает: Решение о безопасности
```

```
Truth Loop
├── Зависит от: Core
├── Используется: Pipeline Executor
├── Проверяет: Утверждения в ответах
└── Обновляет: Knowledge Graph
```

```
Meta Controller
├── Зависит от: Core
├── Используется: Pipeline Executor
├── Управляет: Стратегиями обработки
└── Оценивает: Когнитивную нагрузку
```

```
Event Bus
├── Зависит от: Core
├── Используется: Все компоненты системы
├── Обеспечивает: Внутреннюю коммуникацию
└── Не блокирует: Основной поток выполнения
```

### 2. Память (Memory)

```
RAG Memory
├── Зависит от: Core
├── Используется: Pipeline Executor, Intent Router
├── Хранит: Диалоги и контекст
└── Обеспечивает: Семантический поиск
```

```
Fact Memory

├── Используется: Truth Loop, Knowledge Graph
├── Хранит: Структурированные факты
└── Обеспечивает: Проверку противоречий
```

```
Episodic Memory
├── Зависит от: Core, RAG Memory
├── Используется: Persona, Dreams System
├── Хранит: Эпизодические воспоминания
└── Обеспечивает: Контекст ситуаций
```

```
Semantic Memory
├── Зависит от: Core, Fact Memory
├── Используется: Knowledge Graph, Persona
├── Хранит: Типизированные знания
└── Обеспечивает: Классификацию знаний
```

```
Roots Memory
├── Зависит от: Core, ANTI_DIRECTIVE
├── Используется: Truth Loop, Meta Controller
├── Хранит: Фундаментальные принципы
└── Обеспечивает: Неизменяемую основу
```

```
Persona Memory
├── Зависит от: Core, Episodic Memory, Semantic Memory
├── Используется: Pipeline Executor, Style Manager
├── Хранит: Личность системы
└── Обеспечивает: Персонализацию
```

```
Memory Consolidation
├── Зависит от: Все системы памяти
├── Используется: Dreams System, Health Monitor
├── Объединяет: Разные слои памяти

```

```
Memory Hygiene
├── Зависит от: Все системы памяти
├── Используется: Periodic tasks
├── Очищает: Устаревшую информацию
└── Обеспечивает: Здоровье памяти
```

## 🔄 Циклические зависимости

### Обнаруженные циклы:
1. **Core ↔ Memory**: Core использует Memory, а Memory использует Core
   - Решение: Введение интерфейсов и слабой связанности

2. **Pipeline ↔ Memory**: Pipeline использует Memory, а Memory может триггерить Pipeline
   - Решение: Асинхронная обработка событий

3. **Persona ↔ Memory**: Persona хранится в Memory, а Memory влияет на Persona
   - Решение: Четкое разделение ответственности

## 🧪 Тестирование зависимостей

### Unit тесты
- Каждый компонент тестируется изолированно
- Используются mock-объекты для зависимостей

### Integration тесты
- Тестируются пары связанных компонентов
- Проверяется корректность взаимодействия

### System тесты
- Тестируется вся система как целое
- Проверяется соблюдение зависимостей

## 📈 Управление зависимостями

### Принципы:
1. **Минимизация зависимостей**: Каждый компонент должен зависеть от минимального числа других
2. **Слабая связанность**: Компоненты должны взаимодействовать через интерфейсы
3. **Высокая связанность внутри**: Логически связанные функции должны быть в одном компоненте
4. **Одна точка ответственности**: Каждый компонент отвечает за одну конкретную функцию

### Практики:
1. **Dependency Injection**: Для управления зависимостями
2. **Event-driven**: Для асинхронного взаимодействия
3. **Interface segregation**: Для уменьшения связности
4. **Configuration over convention**: Для гибкости системы
