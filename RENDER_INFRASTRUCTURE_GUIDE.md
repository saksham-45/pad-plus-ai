# 🚀 Render Infrastructure Guide

## 📋 Обзор инфраструктуры

Для устойчивой работы PAD+ AI на Render мы добавили следующие компоненты:

### 🗄️ Базы данных

#### PostgreSQL (Render PostgreSQL)
- **Назначение**: Основная реляционная база данных
- **План**: Free (до 10,000 строк)
- **Использование**: Хранение структурированных данных, метаданных, пользовательских настроек
- **Преимущества**: 
  - Встроенная в Render
  - Автоматическое резервное копирование
  - Высокая надежность
  - Поддержка SQL-запросов

#### Redis (Render Redis)
- **Назначение**: Кэширование и сессионное хранение
- **План**: Free (до 100 MB)
- **Использование**:
  - Кэширование ответов LLM
  - Кэширование RAG запросов
  - Сессионные данные пользователей
  - Rate limiting
  - Очереди задач
- **Преимущества**:
  - Высокая скорость доступа
  - Поддержка TTL (время жизни данных)
  - Поддержка pub/sub для уведомлений

### 🌐 Frontend CDN

#### Static Site + CDN
- **Назначение**: Раздача статических файлов
- **План**: Free
- **Использование**: Статические файлы frontend (CSS, JS, изображения)
- **Преимущества**:
  - Автоматическое CDN
  - Сжатие файлов
  - Кэширование в браузере
  - Высокая доступность

### 📊 Мониторинг и алертинг

#### Встроенная система мониторинга
- **Назначение**: Отслеживание состояния системы
- **Функции**:
  - Сбор метрик CPU, памяти, диска
  - Мониторинг производительности
  - Отслеживание ошибок
  - Контроль кэш hit rate
  - Rate limiting
- **Алерты**:
  - High CPU usage (>90%)
  - High memory usage (>90%)
  - High error rate (>10%)
  - Low cache hit rate (<30%)
  - High response time (>5s)

## 🔧 Конфигурация

### Environment Variables

```bash
# Базы данных
DATABASE_URL=postgresql://postgres:password@db.onrender.com:5432/padplus
REDIS_URL=redis://redis.onrender.com:6380

# Кэширование
CACHE_TTL=3600
SESSION_TTL_HOURS=24

# Rate limiting
RATE_LIMIT_REQUESTS=100
RATE_LIMIT_WINDOW=60

# Frontend
VITE_CDN_ENABLED=true
VITE_API_URL=https://padplus-ai-backend.onrender.com
```

### Health Checks

- **Backend**: `/health` - проверка целостности системы
- **Frontend**: Автоматическая проверка доступности статических файлов
- **Базы данных**: Автоматическая проверка подключения

## 📈 Производительность

### Оптимизации

1. **Кэширование LLM ответов**
   - TTL: 30 минут
   - Снижает нагрузку на LLM API
   - Ускоряет ответы на повторяющиеся запросы

2. **Кэширование RAG запросов**
   - TTL: 10 минут
   - Ускоряет поиск в векторной базе
   - Снижает нагрузку на векторное хранилище

3. **Сессионное кэширование**
   - TTL: 24 часа
   - Хранит состояние пользователя
   - Позволяет поддерживать контекст диалога

4. **Rate limiting**
   - 100 запросов в минуту на IP
   - Защита от перегрузки
   - Предотвращение abuse

### Мониторинг производительности

- **Response time**: Среднее время ответа < 2s
- **Error rate**: Rate ошибок < 5%
- **Cache hit rate**: Hit rate кэша > 50%
- **Uptime**: Доступность > 99%

## 🛡️ Безопасность

### Защита данных

1. **Redis**: Аутентификация по паролю
2. **PostgreSQL**: SSL соединение
3. **Rate limiting**: Защита от DDoS
4. **CORS**: Ограничение доступа по origins

### Резервное копирование

- **PostgreSQL**: Автоматическое резервное копирование
- **Redis**: Persistence (RDB + AOF)
- **Frontend**: Версионирование через Git

## 🚨 Аварийное восстановление

### Сценарии отказов

1. **Отказ Redis**
   - Система продолжает работать без кэширования
   - Автоматическое восстановление при переподключении

2. **Отказ PostgreSQL**
   - Использование fallback режима
   - Локальное хранение данных в файлах

3. **Отказ Backend**
   - Автоматический перезапуск
   - Health checks для обнаружения проблем

4. **Отказ Frontend**
   - CDN обеспечивает высокую доступность
   - Автоматическое восстановление из Git

## 📊 Метрики и мониторинг

### Ключевые метрики

1. **Системные метрики**
   - CPU usage
   - Memory usage
   - Disk usage
   - Network I/O

2. **Прикладные метрики**
   - Request rate
   - Response time
   - Error rate
   - Cache hit rate

3. **Бизнес метрики**
   - Active users
   - Session duration
   - Query patterns
   - LLM usage

### Алертинг

- **Critical**: CPU > 90%, Memory > 90%, Error rate > 10%
- **Warning**: CPU > 70%, Memory > 70%, Error rate > 5%
- **Info**: Cache hit rate < 50%, Response time > 2s

## 🔄 DevOps

### CI/CD

1. **GitHub Actions** (опционально)
   - Автоматический деплой при push в main
   - Тестирование перед деплоем
   - Rollback при ошибках

2. **Render Blueprints**
   - Автоматическое создание инфраструктуры
   - Воспроизводимость окружения
   - Простота масштабирования

### Масштабирование

1. **Horizontal scaling**
   - Render автоматически масштабирует backend
   - CDN масштабирует frontend

2. **Vertical scaling**
   - Возможность увеличения ресурсов
   - Платные планы для высокой нагрузки

## 📞 Поддержка

### Мониторинг состояния

- **Error tracking**: Интеграция с Sentry (опционально)

## 🎯 Рекомендации

### Для production

1. **Перейти на платные планы** при высокой нагрузке
2. **Настроить Sentry** для отслеживания ошибок
3. **Добавить custom domain** для frontend
4. **Настроить SSL сертификаты** (Render делает это автоматически)

### Для разработки

1. **Использовать staging окружение**
2. **Тестировать сценарии отказов**
3. **Мониторить производительность**
4. **Оптимизировать кэширование**
