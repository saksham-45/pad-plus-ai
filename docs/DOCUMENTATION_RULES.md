# PAD+ AI Documentation Rules

Правила поддержки документации в актуальном состоянии.

## 1. README is architecture-driven

Каждый раздел README описывает реально существующий и работающий компонент. Никаких «будет реализовано», «планируется», «в следующей версии» — только то, что есть в текущем релизе.

## 2. No numbers without verification

Любое число в документации должно быть подтверждено из исходного кода:

- количество тестов — `pytest` или `ast`-парсинг;
- количество API endpoints — запуск приложения и подсчёт routes;
- количество фаз pipeline — проверка executor.py;
- количество типов памяти — проверка backend/memory/.

Нельзя писать «145+ endpoints» — только «130+ routes» (после проверки).

## 3. Every architectural statement must reference implementation

Утверждение о наличии компонента должно подкрепляться ссылкой на реализацию:

```
Memory: 6 types implemented in backend/memory/
Pipeline: 22 stages in backend/core/pipeline/phases/
```

Не «у нас есть память», а «6 типов памяти в backend/memory/».

## 4. README describes reality, never intentions

Запрещено:

- `будет добавлено в v4.1`
- `планируется внедрить`
- `в разработке`

Всё это — в раздел Roadmap или в docs/RELEASE_*.md. README содержит только то, что работает сейчас.

## 5. Version must match the release tag

Версия в README (хедер, бейджи) должна совпадать с `git tag`. Если релиз `v4.0.0` — в README `v4.0.0`. Нельзя написать `v4.0` при теге `v4.0.0-beta` и наоборот.

## 6. No dead links

Каждая внутренняя ссылка (на docs/, screenshots/, разделы README) должна резолвиться на момент коммита. Перед релизом — проверка всех ссылок.

## 7. Screenshots must exist in the repository

Скриншоты хранятся в `screenshots/`. Внешние хостинги изображений запрещены — при падении хостинга README теряет иллюстрации.

## 8. One change, one update

При любом изменении кода, которое влияет на:

- количество API эндпоинтов;
- количество тестов;
- количество фаз pipeline;
- количество типов памяти;
- список провайдеров;
- команды запуска;

— README должен быть обновлён в том же коммите.

## 9. Architecture review for README changes

Любое изменение README, затрагивающее архитектурные утверждения, должно быть проверено на соответствие исходному коду. Запрещено менять README «на глаз» без сверки с реализацией.

---

**Нарушение этих правил** — причина отклонить Pull Request до исправления документации.
