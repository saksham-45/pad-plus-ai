# План разработки PAD+ AI - Анализ проекта и стратегия улучшений

> **Примечание:** этот документ сохранён для исторической справки. Актуальная рабочая версия плана доступна в `docs/план_обновленный.md`.

## 1. Философия архитектуры

PAD+ AI представляет собой сложную систему эмоционально-интеллектуального искусственного интеллекта, построенную по модульной архитектуре. Основные принципы:

- **Модульность**: Каждый компонент независим и выполняет конкретную функцию
- **Эмоциональный интеллект**: Система моделирует эмоциональные состояния (Pleasure, Arousal, Dominance)
- **Безопасность**: Многоуровневая система безопасности с анти-директивой
- **Отказоустойчивость**: Graceful degradation при сбоях компонентов
- **Масштабируемость**: Возможность добавления новых провайдеров LLM и функций

## 2. Анализ текущего состояния проекта

### Сильные стороны:
- Комплексная архитектура с модулями: эмоции, память, знания, безопасность
- Поддержка OpenRouter и GigaChat (упрощенная архитектура)
- WebSocket для real-time коммуникации
- Интеграция с Supabase для хранения данных
- Многоуровневая валидация и безопасность

### Проблемные области:
1. **Кодировка и локализация**: Русский текст с проблемами кодировки в логах
2. **Конфигурация**: Сложная настройка .env файлов
3. **Документация**: Недостаточная документация на английском
4. **Тестирование**: Ограниченное покрытие тестами
5. **Остатки LiteLLM**: Необходимо полностью удалить следы убранного LiteLLM
6. **Два провайдера**: OpenRouter и GigaChat нуждаются в отшлифовке и интеграции

## 3. Фазы разработки

### Фаза 1: Очистка и стабилизация (1-2 недели)
**Цель**: Удалить остатки LiteLLM и стабилизировать двухпровайдерную архитектуру

1. **Удаление LiteLLM остатков**
   - Полное удаление всех ссылок на LiteLLM в коде
   - Очистка зависимостей (`requirements.txt`)
   - Обновление конфигурационных файлов
   - Удаление неиспользуемых тестов и компонентов

2. **Стабилизация двухпровайдерной архитектуры**
   - Настройка надежной работы OpenRouter API
   - Оптимизация интеграции GigaChat
   - Создание единого интерфейса для обоих провайдеров
   - Реализация fallback стратегий между провайдерами

3. **Исправление кодировки**
   - Настроить правильную кодировку UTF-8 для всех компонентов
   - Единая система логирования с поддержкой Unicode
   - Перевод всех системных сообщений на корректный русский

### Фаза 2: Улучшение архитектуры (2-3 недели)
**Цель**: Оптимизация производительности и масштабируемости

1. **Оптимизация провайдеров**
   - Кэширование ответов LLM для OpenRouter и GigaChat
   - Балансировка нагрузки между двумя провайдерами
   - Расширенные fallback стратегии при отказах
   - Rate limiting и quota управление

2. **Улучшение системы памяти**
   - Оптимизация RAG (Retrieval-Augmented Generation)
   - Векторная база данных с эффективным поиском
   - Иерархическая организация памяти

3. **Мониторинг и метрики**
   - Расширенные Prometheus метрики для каждого провайдера
   - Health checks для всех компонентов
   - Alerting система с уведомлениями о проблемах провайдеров

### Фаза 3: Расширение функциональности (3-4 недели)
**Цель**: Добавление новых возможностей и улучшение UX

1. **Расширение эмоциональной модели**
   - Более сложные эмоциональные состояния
   - Адаптивное обучение эмоциональных реакций
   - Контекстно-зависимые эмоциональные ответы

2. **Улучшение интерфейса управления провайдерами**
   - Dashboard для мониторинга статуса провайдеров
   - Визуализация использования квот и лимитов
   - Управление API ключами через интерфейс
   - Статистика производительности каждого провайдера

3. **Интеграция с внешними системами**
   - API для сторонних приложений
   - Webhook поддержка для уведомлений о событиях провайдеров
   - Плагинная архитектура для будущего добавления провайдеров

## 4. Компоненты системы (обновленные)

### Core Components:
1. **X-Ray Brain** (`backend/core/xray/brain.py`)
   - Центральный процессор системы
   - Координация всех модулей
   - Принятие решений по выбору провайдера

2. **Emotion Engine** (`backend/emotion/`)
   - Модель PAD (Pleasure, Arousal, Dominance)
   - Эмоциональные состояния и переходы
   - Влияние эмоций на выбор провайдера и ответы

3. **Memory System** (`backend/memory/`)
   - RAG (Retrieval-Augmented Generation)
   - Векторная память (PostgreSQL/pgvector)
   - Иерархическая организация знаний

4. **Knowledge Graph** (`backend/knowledge/`)
   - Граф знаний с отношениями
   - Семантические связи
   - Динамическое обновление

5. **Safety Layer** (`backend/core/safety_layer.py`)
   - Многоуровневая безопасность
   - Анти-директива
   - Content moderation для обоих провайдеров

### Infrastructure:
1. **Provider Management System** (`backend/runtime/`)
   - OpenRouter integration
   - GigaChat integration
   - Унифицированный интерфейс для обоих провайдеров
   - Fallback и балансировка

2. **API Gateway** (`backend/main.py`)
   - FastAPI приложение
   - WebSocket поддержка
   - Аутентификация и авторизация

3. **Database Layer**
   - Supabase (PostgreSQL + pgvector)
   - Redis для кэширования

## 5. Интерфейсы и протоколы

### REST API:
```
GET    /api/v1/health          - Health check (включая статус провайдеров)
GET    /api/v1/mind-state      - Текущее состояние системы
POST   /api/v1/chat           - Чат с системой (с выбором провайдера)
GET    /api/v1/providers      - Статус и статистика провайдеров
POST   /api/v1/documents      - Загрузка документов
GET    /api/v1/knowledge      - Доступ к графу знаний
```

### WebSocket:
```
ws://host/ws
- Подключение: токен авторизации
- Сообщения: JSON формата
- События: emotion_update, memory_update, chat_response, provider_status
```

### Внутренние интерфейсы:
1. **Provider Interface** - Единый интерфейс для OpenRouter и GigaChat
2. **Event Bus** - Pub/Sub для внутренних событий
3. **Dependency Injection** - Управление зависимостями
4. **Pipeline System** - Конвейер обработки запросов с выбором провайдера

## 6. План тестирования

### Модульное тестирование:
- **Coverage goal**: 80% для core компонентов
- **Тестируемые модули**:
  - Provider integrations (OpenRouter, GigaChat)
  - Emotion Engine
  - Memory System
  - Safety Layer
  - Pipeline Components

### Интеграционное тестирование:
- **API Endpoints**: Все REST и WebSocket endpoints
- **База данных**: Миграции и запросы
- **Provider интеграция**: Mock тесты для OpenRouter и GigaChat
- **Fallback тестирование**: Тестирование переключения между провайдерами

### Тестирование безопасности:
- **Input validation**: SQL injection, XSS
- **Authentication**: Token validation, CSRF
- **Anti-directive**: Проверка целостности
- **Provider API key security**: Защита ключей провайдеров

### Performance testing:
- **Load testing**: 100 concurrent users с балансировкой между провайдерами
- **Stress testing**: Предельные нагрузки для каждого провайдера
- **Latency measurement**: Response time под 500ms для обоих провайдеров
- **Fallback performance**: Тестирование скорости переключения при сбоях

## Заключение

Переход с LiteLLM на двухпровайдерную архитектуру (OpenRouter и GigaChat) упрощает систему и повышает ее надежность. Ключевые задачи:
1. Полная очистка от остатков LiteLLM
2. Стабилизация и отшлифовка работы двух провайдеров
3. Создание надежной системы с автоматическим fallback

Упрощенная архитектура с двумя хорошо интегрированными провайдерами обеспечит более предсказуемую работу системы, упростит отладку и снизит операционные риски по сравнению со сложной мультипровайдерной системой через LiteLLM.

План фокусируется на переходе от экспериментальной архитектуры к производственной системе с двумя надежными провайдерами, готовой к масштабированию и дальнейшему развитию.
