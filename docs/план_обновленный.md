# Обновлённый план развития PAD+ AI

> **Текущий рабочий план:** этот документ содержит актуальную версию стратегии развития и состояния проекта на 2026-06-08.

## 1. Введение

Этот документ сравнивает ранее созданный план (`docs/план.md`) с текущим состоянием проекта и предлагает обновлённый план действий.

## 2. Актуальность предыдущего плана

### Что подтверждается проектом
- Проект действительно модульный, с выделенными блоками эмоций, памяти, безопасности и провайдеров.
- OpenRouter и GigaChat являются основными целевыми провайдерами.
- Supabase используется как основная база данных и механизм аутентификации.
- RLS и `user_api_keys` уже рассматриваются как критические зоны.

### Что требует уточнения или уже частично решено
- В проекте уже есть документы и скрипты по RLS: `APPLY_RLS_FIX.md`, `FIX_RLS_README.md`, `check_rls_policies.py`, `QUICK_FIX.md`.
  - Это означает, что проблемы с RLS были зафиксированы и отчасти оформлены, но остаются рабочие проверки.
- В проекте присутствует legacy-файл `backend/runtime/litellm_service.py` и множество ссылок на `litellm` в коде.
  - План по удалению LiteLLM всё ещё актуален: очистка пока не завершена.
- В проекте есть дублирование маршрутов и старые точки входа: `backend/main_stable.py`, `backend/api/routes.py`, `backend/api/frontend_routes.py`.
  - Архитектура нуждается в упорядочивании.

### Что уже не полностью соответствует реальности
- В предыдущем плане было указано, что поддержка OpenRouter и GigaChat уже упрощена и более или менее стабильна.
  - На практике OpenRouter всё ещё ломается: в `chat` запросах не находится ключ, даже если ключ сохранён.
- Тезис о том, что LiteLLM уже убран, не подтверждается: в коде ещё есть активные упоминания `litellm` и legacy-совместимые шины.
- Можно считать, что статус `кодировка/локализация` пока не является главным ограничением, но он остаётся низкоприоритетным техдолгом.

## 3. Текущее состояние проекта

### Ключевые находки
- `backend/api/frontend_routes.py` использует current_user и уже реализует RLS-aware доступ для ключевых CRUD-операций.
- В `backend/core/supabase_client.py` добавлен механизм создания клиента с access token.
- В `backend/runtime/llm_service.py` реализован современный LLM-провайдер для OpenRouter/GigaChat.
- При этом:
  - есть legacy-модуль `backend/runtime/litellm_service.py`;
  - в `backend/core/pipeline.py`, `backend/core/pipeline_handlers.py` и ряде других файлов ещё встречается `litellm`;
  - есть оставшиеся устаревшие маршруты и контроллеры `backend/api/routes.py`, `backend/main_stable.py`.

### Проблемы, требующие немедленного внимания
- `POST /api/v1/chat` с OpenRouter не находит пользовательский ключ и возвращает ошибку `API ключ не настроен`.
- RLS в таблице `user_api_keys` может быть основной причиной, хотя уже есть исправления и документация.
- В `frontend` есть работающий список ключей, но иногда поведение синхронизации и default-ключей остаётся непредсказуемым.

## 4. Новая оценка плана

### Плюсы прежнего плана
- Правильно определены основные проблемы: LiteLLM, провайдеры, RLS, конфигурация, тестирование.
- Подход по фазам (стабилизация → оптимизация → расширение) адекватен.
- Включена логика мониторинга и качества продукта.

### Минусы прежнего плана
- Нет явного шага «удалить legacy маршруты и старые пути запуска».
- Недостаточно внимания к текущей проблеме `chat` + OpenRouter + RLS.
- План слишком общий и не учитывает, что уже есть серьезный набор RLS-документации и промежуточные фиксы.
- Нет явного шага по проверке и рефакторингу `backend/runtime` в сторону единого провайдера.

## 5. Обновлённый план доработки

### Фаза 1. Текущая стабилизация (0-1 неделя)

1. Исправить и стабилизировать доступ к `user_api_keys`:
   - проверить и применить RLS политики в Supabase по `APPLY_RLS_FIX.md` и `FIX_RLS_README.md`;
   - пройти текущие скрипты `check_rls_policies.py`, `check_tables.py`, `check_with_service.py`;
   - убедиться, что `chat`/`chat/stream`/`providers/{provider}/models` используют `get_db_client(current_user)`.
2. Удалить или пометить как deprecated все legacy-модули и маршруты:
   - `backend/runtime/litellm_service.py` — shim, оставить только как совместимость;
   - `backend/api/routes.py` — устаревший LiteLLM-only маршруты;
   - `backend/main_stable.py` — резервный запускной путь, если он дублируется.
3. Привести ключевой provider flow в порядок:
   - проверка `model_preference` / `provider` в `frontend` и `backend`;
   - удостовериться, что OpenRouter ключи читаются и используются.
4. Выполнить smoke-тесты:
   - создание ключа OpenRouter и GigaChat;
   - `GET /api/v1/keys`;
   - `POST /api/v1/chat` с OpenRouter;
   - `GET /api/v1/providers` и `GET /api/v1/providers/{provider_id}/models`.

### Фаза 2. Архитектурная очистка (1-3 недели)

1. Убрать остатки LiteLLM из кода и зависимостей:
   - найти и удалить все упоминания `litellm` там, где уже используется `llm_service`;
   - уменьшить дублирование `get_llm_service` и убрать legacy-сыганы шины.
2. Консолидировать провайдерную логику:
   - сделать единый адаптер/фаçад для OpenRouter и GigaChat;
   - изолировать провайдерные вызовы в `backend/runtime/llm_service.py` и/или отдельный provider manager;
   - унифицировать модельные имена и трансформацию `model_preference`.
3. Упорядочить маршрутные точки и сервисы:
   - оставить один основной FastAPI entrypoint;
   - убрать устаревшие `main_stable` / `api/routes.py` из активного использования;
   - централизовать аутентификацию и авторизацию.
4. Улучшить конфигурацию:
   - документировать `.env` переменные `SUPABASE_URL`, `SUPABASE_KEY`, `SUPABASE_SERVICE_KEY`, `GIGACHAT_AUTH_KEY`;
   - рекомендовать единый файл `.env.example` или скрипт для генерации.

### Фаза 3. Устойчивость и качество (3-6 недель)

1. Тестирование и мониторинг:
   - покрыть ключевые модули unit/integration тестами;
   - добавить тесты для RLS и для пользовательского потокa ключей;
   - настроить health checks и метрики для провайдеров.
2. Производительность и fallback:
   - кэшировать модельные метаданные и статус провайдеров;
   - реализовать fallback OpenRouter → GigaChat для ошибок провайдера;
   - ввести rate limiting и квоты на пользовательские запросы.
3. UI/UX для провайдеров:
   - сделать понятный статус ключа (заполнен/пустой);
   - показать системный GigaChat ключ отдельно;
   - отобразить состояние подключения провайдера и ошибки.

### Фаза 4. Дальнейшее развитие (6+ недель)

1. Добавление новых провайдеров через плагинный интерфейс.
2. Расширение эмоциональной модели и memory/RAG компонентов.
3. Документирование API и внутренней архитектуры для команды.

## 6. Рекомендации по приоритетам

1. Сейчас: RLS + OpenRouter + provider flow.
2. Ближайшие задачи: убрать legacy-lite и упорядочить входные маршруты.
3. Долгосрочно: тестирование, мониторинг, единая provider layer.
